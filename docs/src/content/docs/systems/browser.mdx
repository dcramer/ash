Ash exposes a browser subsystem with a session-oriented API.

## What It Does

- Starts and tracks browser sessions per user.
- Supports deterministic actions: `page.goto`, `page.extract`, `page.click`, `page.type`, `page.wait_for`, `page.screenshot`.
- Stores session metadata and artifacts under `~/.ash/browser/`.
- Exposes control surfaces through the `browser` tool, RPC methods, and `ash browser` CLI.

## Providers

- `sandbox` (default): local Chromium automation via Playwright.
- `kernel`: remote provider adapter for Kernel-backed execution.

## Runtime Integration

Browser is installed through integration composition, not direct ad-hoc wiring in runtime entrypoints.

- Integration: `src/ash/integrations/browser.py`
- RPC methods: `src/ash/rpc/methods/browser.py`
- Tool: `src/ash/tools/builtin/browser.py`

This follows the harness contract in `specs/subsystems.md` and `specs/integrations.md`.

## Configuration

```toml
[browser]
enabled = true
provider = "sandbox"
timeout_seconds = 30.0

[browser.kernel]
api_key = "${KERNEL_API_KEY}"
```

Use `uv run ash doctor` to validate browser config and missing credentials.

For `sandbox` provider, install runtime prerequisites on the host:

```bash
uv sync --all-groups
uv run playwright install chromium
```
